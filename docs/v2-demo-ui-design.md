# V2 Demo UI Design, Without npm or Node.js

Date: 2026-04-23

## Recommendation

For v2, keep the demo as a no-build static browser app:

```text
HTML + CSS + plain browser JavaScript
served by python3 -m http.server
synthetic JSON / JS data only
no npm install
no Node.js runtime
no frontend build step
```

This is the best next UI path for this project because the current demo is already close to that model. Node is currently useful for scripts and tests, but the actual browser screens already run as static HTML, CSS, and JavaScript. V2 should make that strength more intentional: easy to open, easy to explain, safe for synthetic demos, and not tied to a JavaScript build system.

The short answer: use JavaScript, but not another heavy JavaScript framework. Use plain browser JavaScript first. Add a tiny library only if a specific UI problem becomes painful.

## Why Not npm / Node for V2

Avoiding npm and Node makes sense for a clinician-facing demo when the goals are:

- simpler setup on hospital or meeting laptops
- fewer dependency and security conversations
- easier offline rehearsal
- easier review by non-engineers
- less chance that a build tool breaks five minutes before a demo
- clearer boundary that this is a synthetic static prototype, not a production clinical system

The tradeoff is that we should keep v2 disciplined. Without a framework, the UI code needs clear file boundaries and small functions, otherwise plain JavaScript can become hard to maintain.

## V2 Product Shape

V2 should not just be a prettier v1. It should answer a sharper clinical-demo question:

> Can a role-separated previsit workflow help the clinic team see missing information, patient-reported symptom patterns, and answer sources before the physician enters, without implying diagnosis, triage, or treatment advice?

The v2 first screen should be a working demo console, not a landing page. It should immediately let the reviewer pick a synthetic case and move through the role-specific views.

## V2 Screen Concept

### 1. Demo Console

Audience: clinician reviewer, product reviewer, internal demo operator.

Purpose: choose the synthetic case, move through the workflow, and compare what each role sees.

Layout:

- top case selector with four to six synthetic cases
- role tabs: Patient, Nurse, Clinician, Packet, Reviewer
- persistent safety boundary strip
- compact completion state
- source/provenance indicator

Important rule: the demo console can expose role tabs because it is for reviewers. The patient-facing screen itself must still not show staff-only links or labels.

### 2. Patient / Family View

Audience: patient, family/helper.

Purpose: calm, large-button guided intake.

V2 improvements:

- better first step for answer source:
  - 本人回答
  - 本人回答，家屬協助操作
  - 家屬依觀察協助回答
  - 現場人員補問
- one question per screen on mobile
- grouped cards on desktop only when the questions are simple
- always visible choices for 不確定 and 稍後現場再說
- short confirmation page in plain language
- display controls: large text, high contrast, read current question
- no nurse, clinician, reviewer, score, diagnosis, triage, or treatment language

Design focus:

- large touch targets
- calm contrast
- minimal medical terminology
- no dense tables
- source labeling happens quietly in the data, not as scary UI

### 3. Nurse Repair View

Audience: nurse or clinic staff.

Purpose: show only what is missing, uncertain, or source-sensitive.

V2 improvements:

- "ask next" queue instead of a long checklist
- missing item grouped by reason:
  - required core field missing
  - conditional module detail missing
  - family-observed subjective field needs patient confirmation
  - medication list needs support
- each item shows:
  - question to ask
  - why it matters
  - current answer
  - source label
  - mark as asked / skipped / still unknown

Boundary:

- no diagnosis
- no risk level
- no treatment recommendation
- no urgent-care instruction generated by the system

### 4. Clinician Summary View

Audience: physician.

Purpose: a 30 to 60 second scan before the visit.

V2 layout:

- top row:
  - chief concern
  - duration
  - bother score
  - active symptom domains
  - medication readiness
- middle:
  - patient-reported symptom pattern
  - missing or uncertain information
  - source-sensitive items
- right or bottom:
  - source attribution summary
  - copy text
  - print packet

Wording rule:

- Say "patient reported", "family observed", "needs confirmation", or "missing".
- Do not say "likely", "suggests", "high risk", "needs treatment", "recommend", or "diagnosis".

### 5. Visit Packet

Audience: staff demo, print handoff, reviewer.

Purpose: role-separated printable artifact.

V2 improvements:

- one printable packet with three sections:
  - patient confirmation
  - nurse repair log
  - clinician summary
- include timestamp and synthetic-data watermark
- include source labels beside important fields
- include "not a clinical record" boundary

### 6. Reviewer View

Audience: clinical/product reviewer.

Purpose: capture the v2 decision.

Decision choices:

- Continue
- Revise
- Narrow
- Pause

Reviewer evidence fields:

- Would the physician read this?
- Where would this fit in clinic workflow?
- Which fields are useful?
- Which fields are noise?
- What must be removed for safety?
- What should v3 test?

## UI Method Options

### Option A: Static HTML/CSS + Plain JavaScript

Best default for this project.

Use when:

- the demo must run locally with almost no setup
- the data is synthetic
- no real backend is needed
- the UI needs to feel polished and clinical
- the team wants to inspect every file directly

How it runs:

```bash
python3 -m http.server 4173
```

Open:

```text
http://localhost:4173/app/v2-demo/
```

Pros:

- no npm
- no Node
- no build
- works offline if assets are local
- easy for meetings
- good control over patient and clinician UI

Cons:

- code organization requires discipline
- no built-in component model
- no automatic test runner unless we add Python or browser self-checks

Verdict: recommended.

### Option B: Static HTML/CSS + Small Browser Library

Possible libraries:

- Alpine.js for small reactive UI islands
- htmx for server-rendered HTML fragment swaps

Use this only if v2 state handling becomes repetitive.

Important rule: if used, vendor the script locally under the repo instead of depending on a CDN during demos.

Pros:

- still no npm
- easier interactive behavior than manual DOM updates
- small mental model

Cons:

- another dependency to explain
- htmx is most useful when there is a server returning HTML fragments
- Alpine can make complex state harder to debug if overused

Verdict: acceptable fallback, not the first move.

### Option C: Python UI, Such as Streamlit or Gradio

Useful for AI/model exploration, less ideal for this patient/clinician workflow demo.

Use when:

- the main value is backend model behavior
- the team wants quick Python-only experimentation
- visual polish and exact patient-facing interaction are less important

Pros:

- no Node
- fast for Python demos
- easy to connect model calls or data experiments

Cons:

- requires Python packages
- usually less control over a polished clinical workflow UI
- harder to make patient, nurse, and physician surfaces feel distinct
- may feel like an analytics app rather than a clinic workflow

Verdict: good for internal AI experiments, not recommended as the main v2 clinician demo UI.

### Option D: Python Backend + Static Frontend

Use this if v2 needs server behavior but still wants no Node.

Possible backend:

- Python standard library for simple static serving
- Flask or FastAPI only if the team accepts Python package dependencies

Use when:

- synthetic data must be saved across browser sessions
- PDF export needs server rendering
- multiple demo devices need shared state
- htmx-style partial updates become useful

Pros:

- no Node
- more realistic app boundary
- can add server-side logging for demo research

Cons:

- more setup than pure static
- starts to look more like an application than a simple demo
- needs privacy and data-boundary discipline even with synthetic data

Verdict: good v2.5 direction if static v2 proves useful.

## Best V2 Choice

Build v2 as Option A:

```text
static HTML/CSS + plain browser JavaScript + local synthetic JSON
```

Keep Option B in reserve:

```text
Alpine.js only if the patient/nurse/clinician state becomes too awkward
htmx only if we add a Python backend that returns HTML fragments
```

Do not choose Streamlit or Gradio for the main clinician demo unless the demo question changes from workflow usability to AI/model behavior.

## V2 Interaction Pattern

The core v2 interaction should be:

```text
Select synthetic case
-> patient/family intake
-> identify missing and uncertain fields
-> nurse repairs selected fields
-> clinician summary updates live
-> printable packet generated
-> reviewer captures decision
```

The key improvement over v1 is "before and after repair":

```text
Before nurse repair:
  clinician sees missing / uncertain / family-observed fields

After nurse repair:
  clinician sees which items were clarified, by whom, and what remains unknown
```

That gives clinicians a concrete reason to believe the workflow can reduce repeated questions.

## V2 Visual Direction

Patient side:

- warm, calm, high-readability
- large buttons
- single primary action per step
- supportive language
- no clinical dashboards

Nurse side:

- operational and compact
- queue-based
- clear next action
- source warnings visible but not alarming

Clinician side:

- dense but readable
- less decorative
- strong hierarchy
- scan in under one minute
- print-friendly

Reviewer side:

- structured evidence capture
- decision-oriented
- no marketing copy

## Data Model Direction

Move v2 demo content toward JSON files:

```json
{
  "id": "synthetic-frequency-older-adult",
  "label": "晚上常起床尿尿",
  "answers": {
    "filledBy": "patient_self",
    "chiefConcern": "storage",
    "duration": "more_than_1_month"
  },
  "sourceByField": {
    "chiefConcern": "patient_self"
  }
}
```

Benefits:

- easier translation
- easier review by clinicians
- easier screenshot/demo case control
- less coupling between demo content and UI code

Keep the summary logic as pure browser JavaScript functions so the same data can render patient, nurse, clinician, and packet views.

## Verification Without npm / Node

V2 can replace Node-based checks with a lightweight no-Node strategy:

- `app/v2-demo/self-check.html`
  - opens in the browser
  - runs basic assertions
  - checks required DOM elements
  - checks safety wording
  - checks no patient view links to staff pages
- `python3 -m unittest`
  - optional Python checks for JSON validity and required fields
- manual screenshot checklist
  - patient mobile
  - patient desktop
  - nurse repair queue
  - clinician summary
  - print packet

This is not as strong as the current Node tests, but it matches the no-Node constraint and is enough for a synthetic demo cycle.

## Implementation Order

1. Create `app/v2-demo/` with a static demo console.
2. Move synthetic cases into local JSON for v2.
3. Build the patient intake panel with source labeling.
4. Build the nurse repair queue.
5. Build the clinician summary from the same answer object.
6. Add the printable packet view.
7. Add reviewer decision capture.
8. Add `self-check.html` and a small Python JSON validation script if needed.
9. Capture v2 screenshots for the clinician report.

## What V2 Should Prove

V2 should prove three things:

- Clinicians can scan the summary quickly.
- Nurses can repair missing information without diagnosing.
- Patient and family answers remain source-labeled and do not get converted into unsupported clinical conclusions.

If v2 proves those three things, v3 can consider deeper integration questions. If it does not, the project should narrow the clinical scenario before adding more technology.

## Reference Notes

- Python `http.server` can serve a static folder from the command line, but Python documentation warns it is not for production: https://docs.python.org/3/library/http.server.html
- Alpine.js can be included from a script tag, but should be vendored locally for reliable demos: https://alpinejs.dev/essentials/installation
- htmx can be included from a script tag and works best when a server returns HTML fragments: https://htmx.org/docs/
- Streamlit is installed and run through Python tooling, useful for Python demos but not the strongest fit for a polished role-separated clinical workflow UI: https://docs.streamlit.io/get-started/installation
